# TPAC 2025 — Kobe, Japan

> Full-day face-to-face meeting, November 11, 2025.

---

## Overview

**TPAC** (Technical Plenary / Advisory Committee) is W3C's annual flagship conference where working groups, community groups, and the Technical Architecture Group meet in person. In 2025, TPAC was held in **Kobe, Japan**.

The Web Machine Learning Community Group held a **full-day face-to-face session** on November 11, 2025, which was a pivotal moment for WebMCP. The TAG presented feedback, multiple formal resolutions were made, and key architectural decisions were locked in.

## TAG Feedback Presentation

The W3C Technical Architecture Group presented their feedback on WebMCP during the F2F session. TAG members who engaged with the review include:

- **Xiaocheng** ([xiaochengh](https://github.com/xiaochengh)) — Led the review, proposed lower-level primitives approach
- **Marcos Caceres** ([marcoscaceres](https://github.com/marcoscaceres)) — Expressed interest in the review

### TAG's Five Key Points

1. **Motivation is sound**: Frontend agent integration is useful and should be supported by the web platform
2. **Generality**: The API should support different protocols, not just MCP — it should be protocol-agnostic
3. **Privacy & Security**: More exploration needed; the threat model requires further development
4. **Declarative API support**: Particularly useful because MCP might need to interface with the OS statically (without the web app running)
5. **No consensus on approach**: The TAG was split between:
   - **Bottom-up**: Build on lower-level primitives (agent connection, permissions)
   - **Top-down**: Directly introduce a high-level API like `navigator.modelContext`

### The MCP Longevity Concern

The TAG's most provocative comment:
> "MCP is not going to last."

This concern about tying a web standard to a specific protocol led to two responses:
1. Anthropic migrated MCP development to the **Agentic AI Foundation (AAIF)** under the Linux Foundation, establishing broader governance
2. The spec was designed with an SDK-style abstraction layer to decouple from specific MCP versions

## Key Resolutions

Three formal resolutions were made at or immediately following the TPAC session:

### 1. Continue as High-Level API

**RESOLUTION**: Continue development as high-level API per TAG guidance.

Despite the TAG's split opinion, the CG decided to proceed with the high-level approach. Khushal Sagar's argument was decisive:
> "We've already hit that wall. Multiple browsers are building agentic browsing."

The rationale: waiting to build lower-level primitives first would delay practical implementation, and browser vendors were already shipping agentic capabilities.

### 2. Cross-Origin Scoped Out of V1

**RESOLUTION**: Cross-origin agents scoped out of v1.

The question of whether an agent in a parent frame could use tools from a cross-origin child iframe ([Issue #52](https://github.com/webmachinelearning/webmcp/issues/52)) was deferred. The group recognized this was too complex for the first version and would require deeper security analysis.

Discussion moved to [Issue #57](https://github.com/webmachinelearning/webmcp/issues/57) (iframes declaring tools) for a narrower scope.

### 3. Agent Identity Deferred

**RESOLUTION**: Revisit agent identity issue in a different proposal if needed.

Victor Huang's analysis in [Issue #54](https://github.com/webmachinelearning/webmcp/issues/54) identified two classes of identity (opaque and origin-associated), but the group concluded there was no clear need for browser-mediated agent identity in WebMCP v1.

He pointed to the IETF's WebBotAuth working group as the appropriate venue for standardizing agent identity.

## ARIA Reuse Reversal

A significant outcome from TPAC hallway conversations was the **reversal on ARIA attribute reuse** for the declarative API. Brandon Walderman reported:

> "ARIA attributes for browsing agents was discussed at TPAC and the consensus is this is not actually a good idea. One of the concerns was that reusing ARIA could lead to conflicts of interest and possibly incentivize web developers to optimize for machines instead of people using a11y tools."

This decision had been building through PR #26 discussions but was formalized at TPAC. It led to the `tool-*` attribute approach in the declarative API.

## Cross-Group Interactions

### AI Agent Protocol CG

The TPAC session included interaction with the **AI Agent Protocol Community Group**, which works on agent communication protocols. The coordination ensures that WebMCP (browser-to-page interface) and agent protocols (agent-to-agent communication) develop compatibly.

Additionally, a breakout session titled **"AI Agents and web browsers"** explored the intersection of AI agents and the web, covering ongoing work in the Web Machine Learning Working Group and Community Group, and how browsers may need to evolve to support AI-driven applications. The session aimed to identify gaps across W3C groups and inform the scope of a **forthcoming W3C Workshop on AI Agents and the Web** ([announced via public-new-work mailing list](https://lists.w3.org/Archives/Public/public-new-work/2025Jun/0008.html)).

The W3C also published a blog post by **Roy Ruoxi Ran** titled **"AI at TPAC 2025"** ([w3.org/blog/2025/ai-at-tpac-2025/](https://www.w3.org/blog/2025/ai-at-tpac-2025/)) summarizing all AI-related group meetings and breakout sessions at TPAC 2025, including the Web Machine Learning CG's work on WebMCP.

### Apple Feedback

Apple representatives at TPAC provided feedback on API naming:
> "Names should stand the test of tens of years"

This comment influenced the final choice of `navigator.modelContext` — a name designed to be durable even if MCP as a protocol evolves or is replaced.
